# 디자인 시스템

---

디자인 시스템은 <strong>스타일 가이드(Style Guide)</strong>와 <strong>컴포넌트(Components)의 복합체</strong>이며, 컴포넌트들이 함께 사용 될 때 어떻게 사용되어야 하는지, 디자인을 할 때 어떤 규칙들을 지켜야 하는지에 대한 가이드입니다.

<p className="bg-blue-300 p-2 text-dark text-lg font-extrabold"> Style Guide : </p>

특정 브랜드 또는 프로덕트에서 디자인을 할 때 지켜야 하는 규칙들

- 색상
- 아이콘
- 여백 (margin)
- 패딩 (padding)
- 타이포그라피 (typography)
- ...

UI에 있어서는 상당히 추상적인 영역으로 디자인 할 때, 프로젝트의 테마를 만들 때 **전역적**으로 적용되는 부분입니다. 그리고 꼭 UI에만 해당하는 것은 아닙니다. 로고, 굿즈, 문서, 포스터 등에서도 적용 할 수 있습니다.

  ```

#### **2. 적용**

요새는 UI template, tailwindcss, Chakra UI 등에서 폰트, 칼라 팔레트, 간격(margin,padding), container 등을 공통 규격으로 코드로 관리할 수 있도록 도와줍니다.
숙력된 디자이너가 일관적 스타일을 만들고 그 안에서 디자인 해 준 화면과, 미숙한 디자이너가 스타일 없이 화면마다 제각각인 간격 등으로 디자인한 화면은 개발자에게도 고통입니다.
가능한 한 기획 전-중-후 단계에서 디자이너와 협업하여 스타일 가이드를 만들어 프로젝트에 적용하고, 개발 기간 동안 그 가이드 기준으로 디자이너와 소통하는 것이 중요합니다.

- 디자이너가 준 스타일 가이드를 프로젝트에 적용하기 예시 (tailwind css)

## **가장 중요한 것은 문서화 입니다.**

제대로 문서화가 되어있지 않으면 시스템을 사용 할수 없습니다.

아무리, 재사용이 쉬운 컴포넌트를 만들거나 코드 상에 주석을 작성해도,

### 문서화가 되어있지 않다면 그것은 디자인 시스템이라고 부를 수 없는 것입니다.

<br />
<br />

## storybook은 이 작업을 쉽게 하기 위하여 사용하는 도구이며 협업의 채널입니다. (개발자 - 디자이너가 함께 사용합니다).

예를 들면, 다음과 같은 프로세스가 있습니다.

- step1. 디자이너) 스타일 가이드를 제작합니다.
- step2.
  - 디자이너) 스타일 가이드를 기반으로 제플린, adobe XD 등에 컴포넌트, 페이지를 제작합니다.
  - 개발자) 스타일 가이드를 프로젝트의 theme 시스템으로 적용합니다.
- step3.
  - 개발자) 디자이너가 제작한 컴포넌트를, 코드로 작성하여 스토리북에 문서화 합니다.
  - 기획자-디자이너) 스토리북을 통해 컴포넌트를 점검합니다.
- step4.
  - 다른 개발자들은 스토리북을 통해, 컴포넌트 기반의 개발을 합니다.
  - 스토리북으로 목업을 잘 활용하면, 최종 산출물에 대한 검증(E2E 테스트)도 가능합니다.
